# Kartuli Telegram Bot — ТЗ (строго по SRS)

## 0. Цель
Telegram-бот для запоминания грузинской лексики по системе повторений (SRS). Родной язык — русский. Цель — бытовой A1: магазин, транспорт, рынок, small talk с грузинами.

## 1. Подход к обучению
Алфавит **не учится отдельно как голые буквы**. Каждая новая буква вводится через якорное слово, начинающееся с этой буквы (например, `ა — ანანასი / ананас`). Так одновременно запоминается форма буквы, её звук и полезное слово.

Дальше внутри темы вводятся обычные слова и короткие фразы для бытовых сценариев: `grocery`, `transport`, `market`, `small_talk`.

## 2. Жёсткие правила SRS (ядро механики)

Это **единственная** механика повторений. Никаких надстроек поверх метода.

### 2.1 Коробки
| Box | Интервал |
|-----|----------|
| 1   | 1 день   |
| 2   | 2 дня    |
| 3   | 7 дней   |
| 4   | 14 дней  |
| 5   | 30 дней  |

### 2.2 Бинарный исход
У ответа есть только два состояния: `correct` или `wrong`. Никаких `almost_correct` или частичных баллов.

### 2.3 Movement rules
- **Correct** → карточка переходит в следующую коробку (5 → остаётся в 5). `next_review_date = today + interval(new_box)`.
- **Wrong** или нажатие `Не знаю` → карточка падает в Box 1 независимо от текущей коробки. `next_review_date = today + 1`.

### 2.4 Дневная сессия
- Due-очередь = все карточки, у которых `next_review_date <= today`.
- На каждый запуск `/learn` пользователь выбирает дневной кеп `N ∈ {10, 15, 20}` (см. §5).
- Сессия = первые `min(N, len(due-очередь))` карточек из due-очереди в детерминированном порядке: `next_review_date ASC`, затем `current_box ASC`, затем `created_at ASC`.
- Карточки за пределами кепа никуда не «сгорают» — они просто остаются due и автоматически попадают в очередь следующего запуска (см. §2.6).
- Никаких «бонусных» повторов сверх due-очереди.
- **Карточка показывается один раз за сессию.** Если ошиблась — она появится завтра, не сегодня (это и есть смысл Box 1).

### 2.5 Новые карточки
Новые карточки входят в систему как `current_box = 1`. Отдельных квот «N новых в день» не существует — все ограничение задаётся общим дневным кепом из §2.4 поверх due-очереди.

### 2.6 Пропущенный день
Если день пропущен, due-карточки **накапливаются** и ничего не «сгорает». На следующей сессии бот показывает их в обычном порядке (см. 2.4).

### 2.7 Направление как отдельная карточка
Одна лексическая единица (слово или короткая фраза, заучиваемая целиком) даёт **две независимые карточки**:
- `KA → RU` (recognition / распознавание): бот показывает грузинский, ждёт русский.
- `RU → KA` (production / воспроизведение): бот показывает русский, ждёт грузинский.

Каждая карточка имеет собственные `current_box` и `next_review_date` и проходит promotion/demotion отдельно.

**Текущее состояние:** парирование полностью работает для пользовательских карточек через `/add` (см. §6.2). Сидируемая колода `data/decks/a1_seed.json` пока выдаёт только `direction = 'ka_ru'`; автоматическое создание парного `ru_ka` для всей curated-колоды — в roadmap.

### 2.8 Чего НЕ делаем (нарушает SRS)
- Same-session relearn после ошибки.
- Промежуточный статус `almost_correct` с частичным продвижением.
- Лестница подсказок, влияющая на оценку.
- Отдельные квоты «новые vs повторение» (общий дневной кеп из §2.4 — не сюда, он только обрезает уже сформированную due-очередь).
- Weekly checkpoint как отдельный режим оценки.
- Mixed-форматы внутри ревью (выбор из вариантов, мини-диалоги).

Подсказки уровня контента — только пассивная транслитерация под `front`, всегда видимая до ответа. Кнопок-подсказок и интерактивных примеров в сессии нет: они **не меняют** логику оценки и движения по коробкам.

## 3. Card Structure

Логические поля карточки:

| Поле | Тип | Назначение |
|------|-----|------------|
| `id` | int | уникальный идентификатор карточки |
| `lexical_unit_id` | int | связь двух карточек одной лексической единицы |
| `direction` | enum | `ka_ru` \| `ru_ka` |
| `front` | string | то, что показывает бот |
| `back` | string | то, что ожидается от пользователя (canonical) |
| `accepted_answers` | string[] | дополнительные допустимые варианты ответа |
| `letter` | string? | грузинская буква, если карточка алфавитная |
| `transliteration` | string? | транслитерация front (опора для новичка) |
| `example_ka` | string? | короткий пример употребления на грузинском |
| `example_ru` | string? | перевод примера на русский |
| `topic` | enum | категория контента; полный whitelist значений и приоритет ввода — см. §7.1 |
| `current_box` | 1..5 | текущая корзина SRS |
| `next_review_date` | date | когда показать следующий раз |
| `source` | enum | `curated` \| `user` |
| `owner_user_id` | int? | для `source='user'` — владелец карточки; для `curated` — `NULL` |
| `created_at` | datetime | для стабильной сортировки и аналитики |

`transliteration` показывается всегда под `front` до ответа (пассивная подсказка). `example_ka` / `example_ru` хранятся в схеме на будущее, но в текущей версии в UI не выводятся — кнопка-подсказка отсутствует, чтобы не мешать оценке (см. §2.8). Их вывод вернётся, когда seed-колода будет обогащена примерами (см. §8 Skill A roadmap).

### 3.1 Где что хранится (реализация)
В коде поля разнесены по двум таблицам, и это важная часть механики:

- **`cards`** — общий контент карточки: `front_side`, `back_side`, `transliteration`, `example_ka`, `example_ru`, `accepted_answers_json`, `letter`, `direction`, `topic`, `source`, `owner_user_id`, `lexical_unit_id`, `created_at`. Curated-карточки (`source='curated'`, `owner_user_id IS NULL`) видны всем; пользовательские (`source='user'`) — только своему `owner_user_id`.
- **`user_cards`** — per-user SRS-состояние: `current_box`, `next_review_date`, `last_reviewed_at`. Один и тот же curated-контент имеет своё состояние у каждого пользователя.
- **`review_events`** — история исходов (`was_correct`, `previous_box`, `new_box`, `reviewed_at`) для метрик из §10.

Поэтому правила §2 формулируются логически про «карточку», а в SQL изменения идут в `user_cards` (state) + лог в `review_events`.

## 4. Правила проверки ответа (бинарный судья)

Цель — детерминированно вернуть `correct` или `wrong`.

### 4.1 Нормализация ввода
Применяется и к ответу пользователя, и к `back` / каждому элементу `accepted_answers`:
1. Trim пробелов в начале и конце.
2. Схлопывание множественных пробелов в один.
3. Lowercase.
4. Удаление знаков препинания: `. , ; : ! ? "` и кавычек.
5. Замена `ё → е`, `Ё → Е`.
6. Снятие диакритик, не несущих смысл.

### 4.2 Сравнение
Ответ считается `correct`, если выполнено хотя бы одно:
- Нормализованный ввод **точно** равен нормализованному `back`.
- Нормализованный ввод **точно** равен любому элементу нормализованного `accepted_answers`.
- Расстояние Левенштейна между нормализованным вводом и любым из (`back` ∪ `accepted_answers`) ≤ 1 **и** длина целевой строки ≥ 4 символа (это защита от ложных срабатываний на коротких словах).

Иначе — `wrong`.

### 4.3 Кнопка `Не знаю`
Эквивалентна `wrong`. После нажатия бот показывает правильный ответ (`back`) и сообщает, что карточка вернулась в Box 1. Это единственная кнопка, которая видна во время сессии (см. §2.8).

### 4.4 Чего НЕ делаем
Никаких «частично правильно», никаких «угадай ещё раз», никакого автоматического принятия близких синонимов, не указанных в `accepted_answers`.

## 5. Daily Session Flow

1. Пользователь открывает чат и запускает сессию командой `/learn`.
2. Бот формирует due-очередь: все карточки с `next_review_date <= today`, отсортированные по §2.4 (`next_review_date ASC, current_box ASC, created_at ASC`).
3. Бот показывает overview: общее число due-карточек и распределение по Box 1–5. Под overview — три инлайн-кнопки `10`, `15`, `20`. Пользователь выбирает дневной кеп `N`.
4. Сессия = первые `min(N, total_due)` карточек из очереди шага 2. Карточки за пределами кепа остаются due и появятся в следующем запуске. Затем для каждой карточки в сессии:
   1. Заголовок `[i/N] Коробка X`, затем `front`. Под `front` показывается транслитерация, если она заполнена. Никаких других подсказок-кнопок нет (см. §2.8).
   2. Пользователь печатает ответ или нажимает `Не знаю`.
   3. `answer-evaluator-ru-ka` возвращает `correct` или `wrong`.
   4. `srs-engine` обновляет `current_box`, `next_review_date`, `last_reviewed_at` в `user_cards` и пишет запись в `review_events`.
   5. Если `correct` — бот отвечает «Правильно!» и показывает пару `front — back`. Если `wrong` (включая `Не знаю`) — бот показывает `front — back` и сообщает «Карточка вернулась в Box 1». Карточка в этой сессии больше не показывается (см. §2.4).
5. После последней карточки — короткая сводка с метриками сессии:
   - **Повторено** — сколько карточек прошло за сегодня (`COUNT(*) FROM review_events WHERE DATE(reviewed_at)=today`).
   - **Запомнила** — сколько из них с `was_correct = 1`.
   - **Процент** — accuracy за день.
   - **На завтра** — сколько карточек уже стоят с `next_review_date = today + 1`.

   Распределение по коробкам в целом и метрики Box 5 stable доступны отдельно командой `/stats` (см. §10).

## 6. Добавление пользовательских карточек (`/add`)

Сценарий: услышала слово в магазине / на улице, нагуглила перевод, хочешь немедленно занести его в коробки.

### 6.1 Способы ввода
- Команда `/add` с аргументом одной строкой: `ka — ru` или `ru — ka`.
- Команда `/add` без аргументов, после которой бот просит прислать строку в том же формате.
- Бот определяет направление по скрипту: грузинская графика vs кириллица.

### 6.2 Поток
1. Бот определяет язык введённой стороны.
2. Показывает предпросмотр и просит подтвердить.
3. После подтверждения создаёт **две карточки** одной лексической единицы:
   - `KA → RU`: `current_box = 1`, `next_review_date = today + 1`.
   - `RU → KA`: `current_box = 1`, `next_review_date = today + 2`.
   - `source = user` для обеих.

Стартовые даты разведены, чтобы пара не встречалась в одной сессии в первые дни. Дальше каждая карточка живёт по обычным правилам SRS.

### 6.3 Защита от дублей
Если такая лексическая единица уже есть, бот предлагает:
- не добавлять дубликат (текущее поведение).

Примечание: сценарии `Обновить пример / транслитерацию` и `Сбросить обе карточки в Box 1` остаются в roadmap и пока не реализованы.

### 6.4 Roadmap
- Single-side ввод с авто-переводом одной стороны через внешний API (OpenAI / Google / DeepL). Сейчас не реализовано: пользователь вводит обе стороны через тире сам — это предсказуемо и работает оффлайн.
- Авто-обогащение пользовательских карточек примерами / транслитерацией.

## 7. Прогрессия контента

Контент вводится в Box 1 (`source = curated`) силами `vocabulary-curator-a1`. SRS-движок порядок ввода не знает — он работает только с тем, что уже попало в карточки.

### 7.1 Полный список топиков (whitelist)

Топик карточки = ровно одно значение из таблицы ниже. Любой другой топик в seed-файле — баг колоды и должен ловиться валидатором (`src/kartuli_bot/deck_quality.py`).

`alphabet` идёт фоном с самого начала и обязан быть введён не позже первой обычной темы.

### 7.2 Приоритет ввода (волны)

Волна = в каком порядке `vocabulary-curator-a1` вливает новые карточки в Box 1. Волна не обязана закончиться, чтобы началась следующая, но вводится с приоритетом сверху вниз. **Внутри волны топики смешиваются** (см. 7.3) — нельзя «сначала весь `grocery`, потом весь `transport`».

#### Волна 0 — фон
- `alphabet` (вся, идёт параллельно с любой другой волной).

#### Волна 1 — «не потеряться в первый день»
- `numbers` (6–20, затем 30 / 40 / 50 / 100).
- `pronouns_verbs` (я / ты / мой; хочу, могу, иду, делаю).
- `small_talk` (добить базу: «извините», «не понимаю», «медленнее, пожалуйста»).
- `money_bank` (лари, тетри, сдача, чек, банкомат, обмен).

#### Волна 2 — «купить еду и доехать»
- `grocery` (молочка, мясо / рыба, базовые продукты).
- `market` (расширить торг, оплата картой).
- `taxi` (Bolt-фразы: «вы свободны?», «остановите тут», «по навигатору»).
- `transport` (расширить: «следующая», «выхожу», «остановите тут» для маршрутки).
- `food_geo` (хинкали, хачапури, чурчхела, чача, домашнее вино).
- `cafe_restaurant` (заказ, «без острого», «без кинзы», счёт, чаевые).

#### Волна 3 — «выжить, если заболела»
- `body` (части тела).
- `pharmacy` (расширить: симптомы, «у меня болит...»).
- `emergency` (112, «вызовите скорую», «помогите»).
- `colors` (попутно — описать таблетку, цвет машины такси, одежду).

#### Волна 4 — «обустроиться в квартире»
- `housing` (квартира, аренда, депозит, ключ, договор).
- `utilities_repairs` (коммуналка + поломки).
- `telecom` (SIM, тариф, Wi-Fi пароль, «нет сети»).
- `delivery` (Wolt / Glovo: курьер, подъезд, этаж, домофон).

#### Волна 5 — «общаться с людьми»
- `family` (мама, папа, муж, ребёнок).
- `professions` (я работаю..., врач, учитель, программист).
- `hobbies` (спорт, кино, музыка).
- `stranger_chat` (расширить: «давно в Грузии?», «что посоветуете?»).
- `holidays_toasts` (с днём рождения, тост, `გაუმარჯოს`).

#### Волна 6 — «ориентация во времени и в городе»
- `time_calendar` (часы, дни недели, месяцы, «до скольки работает?»).
- `city_navigation` (район, угол, через дорогу, набережная).
- `weather` (дождь, снег, ветер, прогноз).

#### Волна 7 — «ситуативные»
- `clothing` (размер, примерочная, цвет).
- `salon` (постричь, покрасить, маникюр).
- `public_services` (дополнить текущее).
- `travel_geo` (аэропорт, поезд, Батуми, Казбеги).

### 7.3 Правило перемешивания топиков

Контент-курaтор перемешивает топики при сидинге, чтобы пользователю не доставалась «полоса» из подряд идущих карточек одного топика:

- В любой подпоследовательности из 4 подряд новых seeded-карточек — **не более 2** одного топика.
- Внутри одной волны топики идут чередованием, а не блоком.
- На daily-сессию это правило влияет косвенно: порядок очереди диктуется §2.4 (`next_review_date ASC, current_box ASC, created_at ASC`). Перемешивание в очереди достигается тем, что seed-порядок уже перемешан и `created_at` карточек разных топиков чередуется.

## 8. Skills Architecture

Skills — отдельные модули, каждый со своей версией и независимым апгрейдом. Видны в проекте по отдельности (см. `src/kartuli_bot/skills/`).

### Skill A: `vocabulary-curator-a1`
**Зона ответственности:** контент.
- Подбирает слова и фразы по A1 и бытовым темам.
- Формирует пары «буква → якорное слово».
- Готовит карточку с полями `letter`, `front`, `back`, `transliteration`, `example_ka`, `example_ru`, `topic`.
- В целевом дизайне для каждой лексической единицы выпускает две карточки (`KA → RU` и `RU → KA`).
- **Текущее состояние:**
  - Сидируемая колода `data/decks/a1_seed.json` отдаёт только `direction = 'ka_ru'`; парные `ru_ka` карточки для curated-контента — в roadmap.
  - Поля `accepted_answers`, `letter`, `example_ka`, `example_ru` в seed пока пустые/дефолтные; обогащение примерами и алиасами — в roadmap.
  - Для `/add` карточки создаются напрямую через `card-intake` с `source = user` (см. §6); автоматическое контент-обогащение пользовательских карточек — в roadmap.

### Skill B: `srs-engine`
**Зона ответственности:** чистая механика SRS.
- Хранит `current_box`, `next_review_date`, историю исходов.
- Считает `next_review_date` по таблице 1/2/7/14/30.
- Реализует promotion/demotion строго по бинарному исходу (см. 2.3).
- Формирует daily queue по правилу 2.4.

### Skill C: `answer-evaluator-ru-ka`
**Зона ответственности:** бинарный судья.
- Реализует нормализацию (4.1) и сравнение (4.2).
- Возвращает только `correct` или `wrong`.

### Skill D: `session-runner`
**Зона ответственности:** проигрывание сессии.
- Берёт очередь у `srs-engine`, показывает по одной карточке.
- Для каждого ответа вызывает `answer-evaluator-ru-ka`, передаёт исход обратно в `srs-engine`.
- Кнопка `Не знаю` = `wrong`. Не повторяет карточку в той же сессии.

### Skill E: `telegram-ux`
**Зона ответственности:** интерфейс.
- На старте `/learn` — overview due + инлайн-кнопки выбора дневного кепа `10` / `15` / `20`.
- Inline-кнопка в карточке во время сессии: только `Не знаю` (после ответа следующая карточка отправляется автоматически). Транслитерация показывается под `front` пассивно, без отдельной кнопки.
- Reply-команды: `/learn`, `/add`, `/today`, `/stats`, `/settings`, `/reminder_on`, `/reminder_off`, `/skill`.
- Прогресс показывается строго в терминах SRS: distribution по Box 1–5, due-сегодня, due-на-завтра, accuracy за день, Box 5 stable ≥ 30 дней.
- Поток `/add` (см. §6) ведётся через FSM-состояния `waiting_for_pair` и `waiting_for_confirm`.

### Skill F: `card-intake`
**Зона ответственности:** приём пользовательских карточек.
- Реализует поток 6.2.
- Создаёт две карточки в Box 1 со стартовыми датами по 6.2.
- Защита от дублей по 6.3 (сейчас — отказ от создания дубликата).

### Интерфейсы между skills
```
telegram-ux  →  session-runner  →  srs-engine
                     │
                     └──→  answer-evaluator-ru-ka

telegram-ux  →  card-intake  →  vocabulary-curator-a1  →  srs-engine
```

Каждый skill версионируется независимо. Апгрейд одного skill не должен ломать остальных, если соблюдён интерфейс.

## 9. Критерии «курс пройден»

Курс A1 считается пройденным, когда выполнены оба условия:
- ≥ 300 карточек по бытовым темам (`grocery`, `transport`, `market`, `small_talk`) находятся в Box 5.
- Каждая из этих карточек удерживается в Box 5 не менее 30 дней (то есть один полный успешный цикл интервала Box 5).

Алфавит считается освоенным, когда **все** алфавитные карточки (обоих направлений) находятся в Box 4 или выше.

## 10. Метрики прогресса (только в терминах SRS)

- Распределение карточек по Box 1–5 во времени (snapshot в день).
- Доля `correct` на due-карточках по дням.
- Число карточек, удерживающихся в Box 5 ≥ 30 дней.
- Среднее число дней от Box 1 до Box 5 в разрезе `topic`.
- Отдельно те же метрики для `direction = ka_ru` и `direction = ru_ka` (видно, какой навык отстаёт).
